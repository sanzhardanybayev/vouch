# Vouch v1.1 — Design Spec

**Date:** 2026-07-13
**Status:** Approved (pending user review of this doc)
**Builds on:** `2026-07-13-vouch-design.md` (v1, shipped). This spec changes three things surfaced by real-world use; everything else in v1 stands.

## 1. Problems (from real use)

1. **Coverage is misleading.** Folders (`packages`, `cli`, `src`) show 100% when only 1 of 288 files is reviewed. Root cause: `rollup()` skips unreviewed files (`if (!c) continue`), so a folder's % is computed only over the files already reviewed — v1 §8's deliberate "attested-only" choice. A file showing 100% for a whole-file (`kind='file'`) review is correct and stays; the folder aggregation is the bug.
2. **Review metadata isn't reachable from the gutter.** VS Code gutter icons have no hover API, so hovering the ✓ does nothing. The v1 range hover only fires over the reviewed code text, and users don't discover it.
3. **No visible per-engineer view.** The timeline webview has per-user tabs per file, but there is no workspace-level "who reviewed what."

## 2. Decisions

| Question | Decision |
|---|---|
| Coverage denominator | **All git-tracked text files' lines**, not attested-only. Unreviewed file counts as `{reviewedLines: 0, totalLines: N}`. Reverses v1 §8. |
| Per-file coverage | Unchanged: reviewed lines ÷ total lines; `kind='file'` reviewed = 100% (honest). |
| Metadata surface | **CodeLens** above each reviewed span (always visible), links to timeline / re-review / diff. Toggle `vouch.codeLens.enabled` (default true). |
| Per-engineer surface | **"Reviewers" node** at the top of the Vouch sidebar: engineer → review count + lines reviewed, expandable to the files they reviewed. |
| Cost of full line-counting | Bounded: `git ls-files` universe, line-count only (no hash-resolve for unreviewed files), cached by mtime, existing background queue. |

## 3. Coverage engine change

### 3.1 Core (`src/core/coverage.ts`, `treemodel.ts`) — pure

- `fileCoverage` is unchanged.
- **New:** unreviewed files must enter rollups as `{reviewedLines: 0, totalLines: N}`. This is produced in the sidebar layer (needs fs), not core. Core's `rollup` already sums correctly once `{0, N}` entries are passed instead of `null`.
- `treemodel.TreeFile` gains a discriminator so the file *row* can stay dim/no-% while its lines still feed the folder denominator:
  ```ts
  interface TreeFile {
    path: string
    coverage: FileCoverage | null | 'pending'  // {0,N} for counted-but-unreviewed; null only for excluded (binary/empty)
    reviewed: boolean                           // true iff this file has ≥1 current review record
  }
  ```
  - Folder/header rollups use `coverage` (so unreviewed `{0,N}` files drag % down).
  - The file row shows a `%` + colored dot only when `reviewed === true`; a counted-but-unreviewed file (`reviewed === false`, `coverage = {0, N}`) renders dim with **no** description (as today), but is no longer `null`, so it counts.
  - `null` coverage is reserved for genuinely excluded files (binary, empty, unreadable) — excluded from denominators.
- `headerStats.workspacePct` becomes reviewed lines ÷ total lines across **all** counted files. The header string drops "of attested"; new form: `` `${pct}% · ${reviewedFiles}/${totalFiles} files · ${records} reviews` `` where `reviewedFiles` = files with ≥1 review. `attested` field renamed `reviewedFiles` for clarity; `pending` semantics unchanged.

### 3.2 Sidebar queue (`src/vscode/sidebar.ts`)

- The queue currently enqueues **attested** files only. Extend it to enqueue **every** file in `fileList` (all git-tracked). For each:
  - **Reviewed (attested) file:** as today — resolve records (open-doc via pipeline, else disk read text-only) → `fileCoverage` → `{reviewedLines, totalLines}`, `reviewed: true`.
  - **Unreviewed file:** line-count only (no record resolution). New helper `countFileLines(abs): number | null` (null if binary/unreadable/empty). Cache `{mtimeMs, gen, coverage: {0, N} | null, reviewed: false}`.
  - Binary detection: reuse the read; if the buffer contains a NUL byte in the first 8 KB, treat as binary → `null`. Empty (0 lines) → `null`.
- `covCache` value type extends to carry `reviewed: boolean`.
- `treeFiles()`: for every path, read the cache; unreviewed counted file → `{path, coverage: {0,N}, reviewed: false}`; reviewed → `{..., reviewed: true}`; not-yet-computed → `'pending'`.
- Cost note (logged, not silent): the queue now touches all tracked files. Line-counting is O(file size) read + newline count, far cheaper than v1 §8's feared full hash-resolve. mtime cache means steady-state cost is ~0. If a root exceeds 20 000 tracked files, cap enqueues at 20 000 and `log`/status-bar note the cap (mirrors v1's huge-file guard philosophy).

### 3.3 Rendering

- Header: `NN% · R/T files · K reviews` (or `no reviews yet`, or `…` while pending).
- Folder: `NN%` (now honest; a folder with one reviewed file among many reads low).
- Reviewed file: `NN%` + green/yellow/red dot (unchanged).
- Unreviewed counted file: dim `circle-outline`, no description (unchanged look; changed meaning — it now contributes to the parent %).

## 4. CodeLens (`src/vscode/codelens.ts` + `src/core/codelens-text.ts`)

- `src/core/codelens-text.ts` (pure): `codeLensTitle(entries: { authorName: string; status: 'reviewed'|'dismissed'; createdAt: string }[], nowIso: string): string`
  - one author: `✓ Reviewed by <name>, <relTime>` / `⚠ Dismissed (changed since review) — re-review`
  - multiple: `✓ Reviewed by <name> +<k> more, <most-recent relTime>` (or `⚠ Dismissed …` if any current entry is dismissed).
  - Plain string (CodeLens titles are not markdown/HTML — no injection sink; author names are safe as-is). Uses `relTime` from `hovermd`.
- `src/vscode/codelens.ts`: `class VouchCodeLensProvider implements vscode.CodeLensProvider`
  - `provideCodeLenses(doc)`: `pipeline.statusFor(doc)` → for each **distinct anchor** among current records (group by `effectiveRange[0]`), emit CodeLenses at that line:
    1. status lens — command `vouch.openTimeline` with the anchor's representative `recordId`, title from `codeLensTitle`.
    2. `Re-review` lens — command `vouch.reReview` (recordId).
    3. `Diff` lens — command `vouch.showDiff` (recordId), only when status is `reviewed` or the record has a commit.
  - `onDidChangeCodeLenses` fires from `pipeline.onDidUpdate` (matching uri) so lenses refresh when records or text change.
  - Gated by `vscode.workspace.getConfiguration('vouch').get('codeLens.enabled', true)`; when false, return `[]` and don't register churn. Re-read on `onDidChangeConfiguration`.
- `package.json`: add `contributes.configuration` with `vouch.codeLens.enabled` (boolean, default true, description).
- Register provider in `extension.ts` for `{ scheme: 'file' }`.

## 5. Reviewers node (`src/core/store.ts` + `src/vscode/sidebar.ts`)

- `ReviewStore` new method `perEngineer(): { name: string; email: string; reviewCount: number; files: { sourcePath: string; count: number }[] }[]`
  - Derived from `bySource` current records grouped by author email; `reviewCount` = total current records by that author; `files` = per-source counts, sorted by count desc then path. Sorted engineers by reviewCount desc.
  - "Lines reviewed" per engineer needs coverage cross-reference; v1.1 shows **review count + file count** (cheap, from records alone). Lines-reviewed is deferred (§8 backlog) to avoid coupling the Reviewers node to the coverage cache timing.
- Sidebar `Item` union gains `{ t: 'reviewersRoot' }`, `{ t: 'engineer'; email: string }`, `{ t: 'engineerFile'; root: RootEntry; sourcePath: string }`.
  - `getChildren(undefined)` order: `header`, `reviewersRoot` (if any records), then folders/files, then `orphanRoot`.
  - `reviewersRoot` label `Reviewers`, icon `account`. Children: one `engineer` per author, aggregated across roots by email, label `<name>`, description `<reviewCount> reviews · <fileCount> files`.
  - `engineer` children: `engineerFile` per reviewed file, label basename, description `<count>`, dirname as `resourceUri`/tooltip; command `vscode.open` on the file.
- Multi-root: aggregate engineers by email across roots; an `engineerFile` carries its `root` so `vscode.open` resolves the right absolute path. Two roots with the same relative path render as two rows (acceptable; rare).

## 6. Edge cases

- Not a git repo: `lsFiles` empty → `findFallback` (workspace.findFiles, node_modules excluded) supplies the universe. Unchanged.
- Binary/empty/unreadable files: `null` coverage, excluded from denominators and the tree % — never counted as 0-reviewed (would unfairly tank %).
- Huge tracked-file count (>20 000): cap + log.
- CodeLens with `codeLens.enabled=false`: provider returns `[]`.
- Dismissed record: CodeLens shows the ⚠ form; file coverage already excludes dismissed (they're not `reviewed`).
- A file that is both attested and now unreadable: coverage `null` (excluded), `reviewed` flag irrelevant.

## 7. Testing

- **Unit (vitest):**
  - `coverage`/`treemodel`: rollup with `{0,N}` unreviewed entries included → honest folder %; a folder of 1 reviewed (100%) + 3 unreviewed files → correct low %; `null` (excluded) still skipped; header % over all files; `reviewed` flag drives row rendering not rollup.
  - `codeLensTitle`: single/multi author, reviewed/dismissed, relTime.
  - `store.perEngineer`: grouping, counts, sort, multi-author, revoked chains excluded.
- **Integration (`@vscode/test-electron`):**
  - Fixture with one reviewed file among several: folder % is < 100 and matches reviewed-lines ÷ total-lines; header % honest.
  - Reviewers node present with the fixture author and their file.
  - CodeLens provider returns a status lens + re-review lens for the reviewed range; returns `[]` when `vouch.codeLens.enabled=false`.

## 8. Backlog (not v1.1)

- Lines-reviewed per engineer (couple Reviewers node to coverage cache).
- `.vouchignore` to shape the coverage denominator (exclude generated/vendored).
- Per-engineer coverage % of the workspace.
- Gutter icon → clickable (no stable API today; revisit if VS Code adds one).
